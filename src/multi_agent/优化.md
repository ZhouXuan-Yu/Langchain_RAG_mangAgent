## 一、2026 年多 Agent 格局概览

### 三大框架主导生产部署

| 框架 | 核心优势 | 适用场景 | Crayfish 对标 |
|------|----------|----------|---------------|
| **LangGraph** | 图状态机、checkpoint、可回滚 | 复杂编排、合规审查 | 目前已在用（StateGraph） |
| **CrewAI** | 声明式、团队协作、Hierarchical Process | 业务自动化 | 参考 Task/Agent 分离模型 |
| **AutoGen** | 会话式双向通信 | 代码生成、人机协作 | 可引入 UserProxy 模式 |

> LangGraph 2026 年 PyPI 下载量达 3800 万次，为企业级生产部署首选。
> CrewAI 2026 年下载破百万，但 checkpoint 能力弱，适合原型验证。

### Anthropic 2026 Agent 架构 5 核心模式

1. **Prompt Chaining** — 串联步骤，每步输出作为下一步输入
2. **Routing** — LLM 判断类型，分流到专用处理分支
3. **Parallelization** — 多 Agent 并行，结果汇总
4. **Orchestrator-Workers** — ★ 当前系统采用，支持动态任务分解
5. **Evaluator-Optimizer** — ★ 当前"刑曹·质勘 + 吏曹·纠偏"已部分实现

> Anthropic 实测：Orchestrator-Workers 模式下，多 Agent 系统在复杂任务上超越单 Agent 达 90% 以上。
> 子 Agent 运行在独立隔离上下文中，有效防止"上下文腐化（Context Rot）"。

## 三、核心优化方向（按优先级）

### P1-A: Agent 消息总线（AgentMessageBus） ✅ 已实施

**已实现内容**：
- `AgentMessageBus` 单例类（含 `store_result / get_context / mark_done / add_dependency / _has_cycle`）
- `orchestrate()` 中 `bus.reset()` → DAG 构建 → 拓扑排序执行
- `_execute_single_task` 新增 `context` 参数，coder/generic 均支持上下文注入
- `AgentRegistry.dispatch_task` 扩展 `context` 参数，前置结果以结构化 block 注入 prompt

**现状**: 各 agent 独立执行，`all_results` 仅在 orchestrator 内存中，`coder` 任务通过参数接收其他 worker 结果。

**目标**: 构建声明式消息传递层，支持：
- 发布/订阅模式（pub/sub）：一个 agent 完成可通知多个下游
- 依赖订阅：`agent B` 可声明 `depends_on: ["task_A"]`，系统自动注入 A 的结果
- 上下文注入：非 coder agent 也能接收前序结果作为 context

**参考**: LangGraph ACPMessage Schema（Pydantic 严格类型消息）+ LangGraph BusState 架构。

```python
class AgentMessageBus:
    """Structured Message Bus for Agent Communication"""
    _instance: "AgentMessageBus | None" = None

    # 发布订阅表：topic -> [callback]
    _subscribers: dict[str, list[Callable]] = {}

    # 结果存储：task_id -> result
    _results: dict[str, dict] = {}

    def publish(self, topic: str, payload: dict) -> None:
        """发布消息到指定 topic"""
        for cb in self._subscribers.get(topic, []):
            cb(payload)

    def subscribe(self, topic: str, callback: Callable) -> None:
        """订阅 topic"""
        self._subscribers.setdefault(topic, []).append(callback)

    def store_result(self, task_id: str, result: dict) -> None:
        """存储结果并通知下游"""
        self._results[task_id] = result
        self.publish(f"result:{task_id}", result)

    def get_results(self, task_ids: list[str]) -> list[dict]:
        """获取依赖任务的全部结果"""
        return [self._results[tid] for tid in task_ids if tid in self._results]
```

**实施步骤**:
1. 在 `CrayfishOrchestrator.__init__` 中初始化 `AgentMessageBus`
2. `_execute_single_task` 执行完毕后调用 `bus.store_result(task_id, result)`
3. 执行前调用 `bus.get_results(task.depends_on)` 注入 context
4. 非 coder agent 的 `_execute_generic_task` 也支持接收 context 参数

### P1-B: DAG 任务规划（Dynamic Task Graph） ✅ 已实施

**已实现内容**：
- `TaskItem` 新增 `depends_on: list[str]` 和 `execution_mode: str` 字段
- `_create_plan()` prompt 模板新增 `depends_on` / `execution_mode` / `capabilities_needed` 字段说明
- 解析时提取 `depends_on` 和 `execution_mode`，注入 `TaskItem` 构造器
- `orchestrate()` 用 DAG 拓扑排序替代原有并行/顺序分组逻辑（Kahn 算法）

**现状**: Supervisor 规划输出扁平的 task 数组，执行时分 `parallel` / `sequential` 两类。

**目标**: 规划输出支持 DAG（有向无环图），声明：
- 每个 task 的 `depends_on: list[str]`
- 每个 task 的 `execution_mode: "parallel" | "sequential"`
- 环形依赖自动检测与报错

**参考**: CrewAI Task 模型（含 `depends_on`）+ Topological Sort 拓扑排序执行。

```python
class TaskNode:
    task_id: str
    assigned_agent: str
    worker_kind: str
    description: str
    depends_on: list[str] = []        # 前置任务 ID 列表
    execution_mode: str = "parallel"  # parallel | sequential
    status: str = "pending"

    # 拓扑排序：计算入度，只执行入度为 0 的任务
    # 每完成一个，递减其下游的入度，触发新任务就绪
```

**改造点**:
- `_create_plan()` prompt 新增 `"depends_on"` 和 `"execution_mode"` 字段说明
- `_create_plan()` 解析时构造 `TaskNode` 而非扁平的 `TaskItem`
- `orchestrate()` 中用拓扑排序替代当前的两类分组逻辑
- 检测循环依赖：`graph.add_edge(a, b)` 时检查是否存在 `b → ... → a` 路径

### P1-C: 能力声明式路由（Capability-Based Routing） ✅ 已实施

**已实现内容**：
- `agent_profiles` 表新增 `capabilities TEXT DEFAULT '[]'` 列（兼容迁移）
- `AgentRegistry.create()` / `update()` / `_row_to_dict()` 均支持 `capabilities` 字段
- 三条内置 Worker 种子数据写入默认 capabilities（search/browse_page、memory/knowledge_search、code_generation）
- `_create_plan()` 构建 agent 描述段落时注入 `[能力: xxx]` 标签
- `SUPERVISOR_PROMPT_TEMPLATE` 新增 `capabilities_needed` 字段，引导 Supervisor 做 capability 匹配

**现状**: 路由依据 `worker_kind` 硬编码，`enabled_kinds` 是字符串集合。

**目标**: 每个 agent 声明 `capabilities: list[str]`，Supervisor 根据任务需求的 capability 标签匹配最合适的 agent。

```sql
-- agent_profiles 新增字段
ALTER TABLE agent_profiles ADD COLUMN capabilities TEXT DEFAULT '[]';
-- 示例值: '["web_search", "code_generation", "data_analysis"]'
```

**Supervisor 匹配逻辑**:
```
task 需要 capability ["web_search", "code_generation"]
→ 匹配同时拥有这两个的 agent（最优）
→ 否则匹配拥有最多重叠的 agent（次优）
→ 若无匹配，回退到 worker_kind 兜底
```

**实施步骤**:
1. `AgentRegistry.update()` 支持 `capabilities` 字段
2. `_create_plan()` prompt 传入所有 agent 的 `capabilities` 描述
3. 解析时用 capability overlap 计算匹配度，替换纯 worker_kind 过滤
4. 前端 agent 编辑器增加 capability 标签选择 UI

### P2-A: 长期记忆与跨会话状态（Cross-Session Memory）

**现状**: 记忆仅来自 RAG Worker 的向量检索（`ChromaMemoryStore`），单会话有效。

**目标**: 实现四层记忆架构（参考 2026 Agent Memory 最佳实践）：

| 层级 | 名称 | 生命周期 | 技术选型 |
|------|------|----------|----------|
| L1 | Session Memory | 单次编排 | 当前 `all_results` 内存 |
| L2 | Working Memory | 单次编排，Agent 间共享 | `AgentMessageBus._results` |
| L3 | Episodic Memory | 跨会话，事件序 | SQLite `agent_episodes` 表 |
| L4 | Semantic Memory | 跨会话，向量检索 | ChromaDB（已有） |

**L3 Episodic Memory 设计**:
```python
# 每次编排完成后，自动记录 episode
CREATE TABLE agent_episodes (
    id TEXT PRIMARY KEY,
    job_id TEXT,
    requirement TEXT,
    tasks JSON,         -- 任务清单及结果
    quality_score REAL,
    duration_ms INTEGER,
    created_at TEXT
);
```

**前端增强**: 记忆面板可查看历史 episode，按质量评分排序，点击回放任务执行过程。

### P2-B: MCP 协议集成（Model Context Protocol）

**现状**: 工具通过 `bind_tools` 硬编码绑定，工具集固定。

**目标**: 引入 MCP（Model Context Protocol），实现：
- 工具动态发现：`GET /mcp/tools` 返回可用工具列表
- 工具按需绑定：编排时根据 task capability 动态加载工具集
- 跨框架互操作：其他 MCP 客户端（如 Cursor）可直接调用 Crayfish 工具

**参考**: Anthropic 主导，2025 年底捐赠给 Linux Foundation，现有 9700 万月 SDK 下载量，2000+ 服务器注册。

```
MCP Server (d:\Aprogress\Langchain)
    ├─ Tools: web_search, browse_page, memory_search, code_execute, ...
    ├─ Resources: agent_profiles, task_history, episodes
    └─ Prompts: supervisor_planning_prompt, worker_execute_prompt

MCP Client (Cursor Agent / Claude Desktop)
    └─ 通过 SSE 或 stdio 连接，发现并调用 Crayfish 工具
```

**实施步骤**:
1. 实现 MCP 服务器端（参考 `fastmcp` 或手动实现 SSE 端点）
2. 工具注册中心 `ToolRegistry`：工具声明名称、描述、参数 schema、所属 capability
3. `GenericWorker.execute()` 改为动态查询 `ToolRegistry`，绑定相关工具
4. 前端 MCP 市场页面：浏览/搜索/启用第三方 MCP 服务器

### P2-C: 可观测性与成本追踪（Observability）

**现状**: 无追踪基础设施，仅有 Python `logging`。

**目标**: 接入 LangSmith 或自建 OpenTelemetry：

| 维度 | 实现方案 |
|------|----------|
| 分布式追踪 | LangSmith `Client` 或 OpenTelemetry Span，每个 agent 调用生成一个 span |
| Token 计数 | `TokenTracker`（已有）入库，按 job_id 聚合 |
| 成本追踪 | 每个 LLM 调用记录 `provider/model` + `input_tokens` + `output_tokens` + `cost_usd` |
| 审计日志 | 每个编排 job 的 `agent_episodes` 持久化，含完整事件序列 |
| 监控面板 | 前端新增"编排统计"页面：平均质量分、平均耗时、平均 token 消耗 |

### P2-D: 多级编排（Multi-Level Orchestration）

**现状**: Supervisor 一次性规划全部子任务，无层级。

**目标**: 支持"总任务 → 子任务 → 孙任务"的多级拆分：
- 顶级 Supervisor（总智能体大哥）拆分大任务
- 子任务若仍复杂，递归调用第二级 Supervisor
- 最多支持 3 层（防止无限递归）

**参考**: CrewAI 的 `Hierarchical Process`，一个 Crew 可作为另一个 Crew 的 Agent。

```python
class HierarchicalOrchestrator:
    """支持多级编排"""
    MAX_DEPTH = 3

    async def orchestrate_recursive(
        self,
        requirement: str,
        enabled_agents: list[str],
        depth: int = 0,
    ) -> dict:
        plan = await self._create_plan(requirement, enabled_agents)
        results = []
        for task in plan:
            if depth < self.MAX_DEPTH and self._is_complex(task):
                # 递归拆分复杂子任务
                sub_result = await self.orchestrate_recursive(
                    task.description, enabled_agents, depth + 1
                )
                results.append(sub_result)
            else:
                results.append(await self._execute_single_task(task))
        return self._aggregate(results)
```
