## 一、整体架构

```
Frontend (orchestrate.html)
    ↓ POST /orchestrate
API (api.py → orchestrate endpoint)
    ↓ BackgroundJobManager.create_job()
    ↓ asyncio.create_task(_run_orchestrator_job)  ← 后台异步
        ↓
    CrayfishOrchestrator.orchestrate()
        ├─ Step 1: _create_plan()        ← 中书省·制敕（Supervisor LLM 规划）
        ├─ Step 2: _execute_single_task() ← 尚书省·牒发六曹（执行分发）
        ├─ Step 3: _evaluate_quality()    ← 刑曹·质勘
        ├─ Step 4: self_heal 循环         ← 吏曹·纠偏
        └─ Step 5: _build_final_result()  ← 台阁·进呈
```

- `src/graph/orchestrator.py` — CrayfishOrchestrator，核心编排器

## 二、当前已知缺陷

### 缺陷1: Supervisor 规划硬编码 worker_kind 白名单（严重）

**位置**: `src/graph/orchestrator.py` `_create_plan()` 方法

**现象**: 新增 agent 后，即使在 `enabled_agents` 里指定了，Supervisor 规划阶段也可能将其过滤或无法正确路由。

**根因**: `_short_map` 只映射 `{"search", "rag", "coder"}` 三个简写；`enabled_kinds` 集合用于校验，但前端 `OrchestrateRequest.enabled_agents` 默认值也是 `["search", "rag", "coder"]`。如果用户传了自定义 agent_id，需要确保：
1. 该 agent 在 `AgentRegistry` 中有 `worker_kind` 字段
2. `enabled_agents` 里传入的是 agent_id 或对应的简写

**验证方法**: 检查 `participants` 参数中是否包含自定义 agent 的完整档案（id, name, worker_kind, description）。

---

### 缺陷2: 执行器硬编码 — 自定义 Agent 无专门执行器（已修复）

**位置**: `src/graph/orchestrator.py` `_execute_single_task()` 方法，第 455-466 行

**修复方案**: 已实现 `_execute_generic_task()` 方法，替代原 `_dispatch_generic()`：

- 实时推送三阶段 SSE `worker_progress`（初始化 → 分发 → 完成）
- 新增 `_evaluate_generic_quality()` 启发式质量评分
- 调用点已改为 `await self._execute_generic_task(task, progress_callback)`

---

### 缺陷3: 前端只渲染内置 Agent 步骤（已修复）

**位置**: `pages/orchestrate.html` `buildStepList()` 函数

**修复方案**:
- `buildStepList` 中增加动态杂曹槽位：从 `participants` 遍历，跳过内置 3 个，其余渲染为"六部 · 杂曹"
- 新增 `_resolveStepKey()` 辅助函数：通过 `agentId` 在 `_steps` 映射中查找，解决 SSE 事件 `agent` 字段是 `agent_id` 而非 `worker_kind` 的不匹配问题
- `onWorkerStart` / `onWorkerDone` / `onWorkerProgress` / `onWorkerRejected` 均已改用 `_resolveStepKey`

---

### 缺陷4: Agent 间无状态共享机制

**现象**: 各 agent 独立执行，结果不互通。如果自定义 agent A 的输出是 agent B 的输入，无法表达依赖。

**当前**: 只有 `coder` 任务接收 `context = all_results` 作为前序上下文。

**修复方向**: 引入**消息总线 / 共享状态**：
```python
class AgentMessageBus:
    """各 Agent 之间的消息传递与结果共享"""
    results: dict[str, Any]  # task_id -> result
    
    def publish(self, task_id: str, result: dict)
    def subscribe(self, task_ids: list[str]) -> list[dict]
```

---

### 缺陷5: 规划能力受限

**位置**: `src/graph/orchestrator.py` `_create_plan()` prompt，第 354 行 `tasks[:3]`

**限制**:
- 最多 3 个子任务
- 无子任务依赖关系声明
- 无并行/顺序执行标记

**修复方向**:
- 支持 `tasks` 数组动态扩展（>3 时分批编排）
- 每个 task 增加 `depends_on: list[str]` 字段
- 每个 task 增加 `execution_mode: "parallel" | "sequential"` 字段

## 三、优化路线图

### Phase 1: 让自定义 Agent 能参与编排（P0）✅ 已实现

1. **GenericWorker 执行器**
   - 方法：`_execute_generic_task()` + `_evaluate_generic_quality()`
   - 支持实时 SSE 推送（初始化 → 分发 → 完成）+ 启发式质量评分

2. **前端动态步骤**
   - `buildStepList()` 从 `participants` 动态构建杂曹槽位
   - `_resolveStepKey()` 解决 agent_id ↔ worker_kind 路由映射

3. **Supervisor 动态规划**
   - `enabled_kinds` 允许自定义 `worker_kind` 通过（仅内置才做严格校验）
   - `_fallback_plan()` 遍历 `self._participants` 动态构建 kind-to-id
   - 移除 `tasks[:3]` 上限，prompt 调整为"合理拆解，保证完备性"

### Phase 2: Agent 协作与状态共享（P1）

3. **AgentMessageBus** — 结果共享
   - 各 worker 执行完毕后，结果写入共享总线
   - 后续 worker 可引用前序结果作为 context
   - 支持 `depends_on` 声明依赖

4. **动态树形规划** — Supervisor 增强
   - 支持 DAG 依赖 + 执行模式
   - 支持多级编排（父任务 → 子任务）

### Phase 3: Agent 能力注册与智能匹配（P2）

5. **Capability-Based Routing** — 超越 worker_kind
   - 每个 agent 声明 `capabilities: list[str]`（如 `["web_search", "data_analysis", "code_generation"]`）
   - Supervisor 根据任务需求匹配最合适的 agent
   - 不再依赖固定的角色分配

6. **Agent 生命周期管理**
   - 支持 agent 动态注册/注销
   - 支持 agent 能力热更新
   - 编排时动态发现可用 agent

## 四、参考模式

### CrewAI 模式（推荐参考）

```
Crew
 ├─ Agents (llm, role, goal, backstory, tools, allow_delegation)
 ├─ Tasks (description, agent, expected_output, async_execution)
 └─ Process (hierarchical | parallel)
     └─ ForemanManagerAgent → 委托任务给 agents
```

关键思想：
- **Agent 是能力载体**：声明式工具集，而非硬编码执行器
- **Task 是工作单元**：包含 agent 绑定 + 依赖 + 输出格式
- **Process 是编排策略**：支持层级（Supervisor 分配）和并行

### LangGraph Supervisor 模式（已在用）

```
Supervisor (LLM with routing tools)
 ├─ Member1 → 执行子任务 → 返回结果
 ├─ Member2 → 执行子任务 → 返回结果
 └─ Supervisor → 聚合结果 → 最终回复
```

当前 Crayfish 已部分实现，但 Supervisor 规划能力需增强。

## 五、数据模型

```sql
agent_profiles {
  id           TEXT PRIMARY KEY,      -- agent_xxx
  name         TEXT,                  -- 显示名
```
